# 0003. 반응형 전략과 모바일 메타포

- 상태: 채택
- 날짜: 2026-06-22
- 관련: [[0002-nextjs-conversion]]

## 배경

RueHanix는 데스크톱 타일링 WM(Hyprland 모사)이다. 창 6개 타일링, 드래그 거터,
키보드 단축키(Super+…), 데스크톱 위젯(neofetch/sysmon)이 모두 **넓은 화면·마우스·키보드**를
전제한다. 좁은 화면에 그대로 올리면 쓸 수 없다. 요구: 반응형으로 만들되 "OS 스타일이
너무 이상해지지 않는 선". 사용자가 진행 전 3개 정책을 확정했다.

## 결정

**단일 경계 768px**. `lib/ruehanix/responsive.ts`의 순수 함수 `isMobileWidth(W)`로 판정(폭 기준,
포인터 종류 무관 — 넓은 터치스크린도 데스크톱 타일링 유지).

- **≥768 (태블릿·데스크톱)**: 기존 타일링 WM 그대로. 변경 없음.
- **<768 (모바일)**: 폰 OS 메타포로 전환.
  - 포커스된 앱 **하나만 풀스크린**(상단바 아래 ~ 하단 독 위). 나머지 창 숨김, 타일링·거터 없음.
  - **하단 독**: 8개 앱 아이콘, 탭으로 풀스크린 전환, 활성 앱 강조(가로 스크롤).
  - **상단바 간소화**: ruehanix(홈) + 포커스 앱 이름 + 배터리/시계. 워크스페이스·시스템 칩 제거.
  - **홈 화면**: 포커스 앱이 없으면 아바타+이름 카드(폰 홈스크린 느낌).
  - 데스크톱 위젯·런처 오버레이·키바인드 오버레이·드래그 거터 **비활성**.
  - 부팅 시퀀스·테마 엔진·8개 앱 콘텐츠는 **그대로 유지**.

순수 레이아웃 함수(`mobileAppRect`, 상수 `MOBILE_TOPBAR=40`, `MOBILE_DOCK=64`)는 vitest로 테스트.
뷰모델이 `isMobile`로 분기해 타일/거터/독/홈을 산출하고, 셸이 조건부 렌더한다.

## 이유와 대안

- **세로 스택(대안)** — 창을 세로로 쌓기. 타일 느낌은 남으나 좁은 폭에서 좌우가 답답하고
  여전히 "데스크톱 흉내"라 폰에서 어색. 기각.
- **데스크톱 권장 + 콘텐츠 폴백(대안)** — 모바일에서 OS 메타포를 버리고 글 리스트만 표시.
  구현은 쉬우나 제품 정체성(데스크톱 셸)을 모바일에서 통째로 포기. 기각.
- **채택: 풀스크린 앱 + 하단 독** — 폰 사용자가 이미 아는 메타포(앱 스위처/독)로 자연스럽고,
  데스크톱 WM 메타포와도 연속성이 있다("창 = 앱"). "너무 이상하지 않은 선"에 부합.
- **포인터 기준(pointer:coarse) 대신 폭 기준** — 넓은 터치 모니터에서도 타일링이 유용하다는
  사용자 판단을 반영. 단순하고 SSR 친화적(폭은 resize로 추적, 포인터 미디어쿼리 불필요).

## 영향

- 데스크톱 동작은 무변경(회귀 위험 격리). 모바일은 같은 상태(open/focused/ui)를 공유해
  리사이즈로 경계를 넘으면 자연스럽게 모드 전환.
- 키보드 핸들러는 모바일에서도 attach되나 런처/키 오버레이를 폭으로 가드해 노출 안 됨.
- 향후: 모바일 제스처(스와이프로 앱 전환), 독 길게 눌러 닫기, 안전영역(safe-area-inset) 대응이 후보.
