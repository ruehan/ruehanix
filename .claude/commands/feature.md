---

작업: $ARGUMENTS

CLAUDE.md의 역할·워크플로와 CONVENTIONS.md의 표준을 따른다.

1. 상황을 먼저 판단한다 — 새 프로젝트(빈 디렉터리)인지, 기존 프로젝트인지, 단발성 작업인지.
   - 새 프로젝트: CONVENTIONS.md 구조(FSD / 도메인 모듈) + 검증 인프라 세팅 + git 초기화.
   - 기존 프로젝트: 레포 인벤토리를 먼저 하고 결과를 보고한다. HARNESS GAP이 있으면 명시한다. 기존 패턴을 우선한다.
2. 진행에 필요한 질문이 있으면 **여기서 모아 한 번에** 묻는다. 이후에는 자율 진행하며,
   사소한 모호함은 합리적 가정으로 채운다(중요한 가정은 ADR로 남긴다).
3. 작업용 feature 브랜치를 만든다 (`feat/…` 등).
4. 요구사항을 한 문장 성공 기준으로 재진술하고, 만들거나 수정할 것을 짧게 제시한다.
5. TDD로 구현한다 — Red → Green → Refactor. 의미 있는 단계마다 커밋한다
   (한국어 Conventional Commits, 각 커밋은 green 상태, 도구 서명·Co-Authored-By 금지).
   중요한 결정을 내리면 그 자리에서 `docs/decisions/NNNN-제목.md`에 ADR로 기록한다.
6. `bash ~/.claude/scripts/verify.sh`로 자체 검증한다. 실패하면 통과할 때까지 수정한다.
7. `code-reviewer` 서브에이전트에게 변경 요약과 함께 핸드오프한다.
   라운드별 판정·지적사항·반영 내용을 기억해 둔다.
8. 리뷰 판정이 "수정 필요"면 지적사항을 반영하고 6번부터 다시 한다.
   최대 3회 반복 후에도 통과하지 못하면 멈추고 사용자에게 상황을 보고한다.
9. 리뷰 통과 후 작업 브랜치를 `main`으로 병합하고 feature 브랜치를 삭제한다.
10. 기록을 남긴다.
    - `docs/reviews/YYYY-MM-DD-작업제목.md` — 리뷰 라운드별 판정·지적사항·반영 내용 (형식은 CONVENTIONS.md).
    - `docs/worklog.md` 맨 위에 이번 작업 항목 추가 (리뷰 줄은 요약 + 리뷰 기록 파일 링크).
11. 변경 요약 + 검증 결과 + 리뷰 판정 + 브랜치/커밋 내역 + 남긴 기록(ADR·리뷰 파일 등)을 보고한다.
    `git push`가 필요하면 사용자 승인을 요청한다 — 자동 실행하지 않는다.

가드레일에 해당하는 작업(푸시·배포·마이그레이션·삭제 등)은 절대 자동 실행하지 않고 사용자 승인을 받는다.
